# Design Guidelines

## Design Context

### Users

WebAssembly developers building with Components and WIT interfaces. They use
`wasm-cli` from the terminal to install, publish, and run Wasm packages — and
browse the web registry to discover what's available. Their context is
development workflow: fast iteration, dependency management, and composing
components. The job to be done is **finding, installing, and shipping Wasm
packages with zero friction**.

### Brand Personality

**Minimal, Clean, Professional.**

The tone is direct and technical — like well-written documentation. No marketing
speak, no unnecessary flair. Information density is preferred over decoration.
The tool should feel like a sharp, reliable instrument that stays out of the way.

### Aesthetic Direction

- **Visual tone**: Editorial minimalism. Calm paper canvas, near-black ink,
  generous whitespace, restrained pastel accents. Documentation-grade.
- **Reference**: Editorial design systems (Linear, Vercel docs, Notion's quieter
  surfaces). Soft pastel chips against neutral paper.
- **Anti-reference**: Neon dashboards, candy-bright SaaS palettes, gradient-heavy
  hero sections, marketing-flash interfaces.
- **Emotional goal**: Composure — "this feels considered, calm, and quick."
- **Theme**: Light **and** dark, both first-class. System preference by default,
  with a manual toggle persisted in `localStorage`. Each token is tuned per
  theme — not algorithmically inverted. CLI/TUI keep standard terminal palette.

### Color System

Paper-and-ink core with pastel categoricals. CSS variables define every
token; Tailwind reads them via `var()`. Two themes share the same token names.

**Core neutrals**

| Token        | Light       | Dark        | Usage                                    |
|--------------|-------------|-------------|------------------------------------------|
| `canvas`       | `#F4F4F5`   | `#1C1C20`   | Page background                          |
| `surface`      | `#FFFFFF`   | `#26262B`   | Cards, panels, input backgrounds         |
| `surfaceMuted` | `#E8E8EA`   | `#2F2F35`   | Inline code chips, hover, subdued bands  |
| `ink-900`      | `#18181B`   | `#ECECEE`   | Body text, headings, primary actions     |
| `ink-700`      | `#3F3F46`   | `#B5B5BB`   | Secondary text                           |
| `ink-500`      | `#71717A`   | `#8B8B92`   | Tertiary text, metadata, mono labels     |
| `ink-400`      | `#A1A1A8`   | `#76767D`   | Muted detail (~4.0:1; non-essential)     |
| `ink-300`      | `#D4D4D8`   | `#4A4A50`   | Disabled, very faint marks               |
| `line`         | `#E4E4E7`   | `#3A3A40`   | Default borders, swatch outlines         |
| `lineSoft`     | `#EFEFF1`   | `#323238`   | Internal subdivisions (table rows)       |
| `rule`         | `#18181B`   | `#6B6B72`   | Editorial 1.5px section dividers         |

**Semantics** (used for text far more than fills; tuned restrained, not neon)

| Token      | Light     | Dark      | Usage                                  |
|------------|-----------|-----------|----------------------------------------|
| `positive` | `#1F8A4C` | `#5EC787` | Success deltas, confirmations          |
| `negative` | `#9B4F5E` | `#EE7B8E` | Errors, destructive actions, deltas    |
| `accent`   | `#18181B` | `#8FB1F5` | Focus rings, links-on-dark, selection  |

**Categoricals** — 10 distinct pastels for charts, labels, status chips, tags.
Light mode uses soft low-chroma pastels (~0.04). Dark mode uses higher-chroma
variants (~0.10) so hue identity carries against the dark canvas. Each pastel
ships with a paired `*Ink` color **designed to clear 4.5:1 on its fill** —
verify with a contrast tool when introducing new combinations.

```
blue   pink   green   peach   lilac
cream  teal   rust    plum    slate
```

Approx hue spread: pink ≈10–22° (warmer in dark), rust ≈50°, peach ≈75°,
cream ≈100°, green ≈148°, teal ≈215°, blue ≈264°, lilac ≈287°,
plum ≈302–322° (cooler/more violet in dark), slate = neutral gray. Hues are
tuned per theme rather than algorithmically inverted. Pick categoricals by
index, not by meaning.

**CLI colors**: Green (success/checkmarks), Yellow (active/progress bars),
Dim (secondary info like versions and sizes).

**TUI colors**: Yellow (active state, headers), Cyan (titles, counts),
Dark Gray (help text, disabled), Magenta/Blue/Green (WIT syntax highlighting).

### Typography

- **Font family**: Iosevka Web (self-hosted woff2) is the canonical face for the
  production web frontend — four weights: Regular (400), Medium (500),
  SemiBold (600), Bold (700). The reference design system page (`design-system.html`)
  uses the system sans stack (`-apple-system`, `BlinkMacSystemFont`, `system-ui`,
  ...) for portability; product pages should prefer Iosevka. Mono uses
  `ui-monospace` family stack everywhere.
- **Weight strategy**: Body uses `font-normal` (400). Labels/emphasis use
  `font-medium` (500). Headings use `font-semibold` (600) with tight
  `tracking-tight`. Reserve heavier weights for rare, intentional emphasis.
- **Size scale**:
  - `text-[24px]` h2 section titles
  - `text-[16px]` h3, card titles
  - `text-[14px]` h4, body, primary content
  - `text-[13px]` secondary body, descriptions
  - `text-[12px]` mono labels, eyebrow text (`uppercase tracking-wider`)
  - `text-[11px]` smallest mono detail, badges
- **Mono identity**: Package names (`wasi:http`), commands (`wasm install`),
  versions, IDs, paths, durations, and inline code use the mono stack. Inline
  code in prose is wrapped in a `<code>` chip with `bg-surfaceMuted` background
  and `text-[0.875em]` so it reads as a token, not a brightness jump.
- **Rendering**: `antialiased` on body for crisp glyph rendering.

### Form Controls

Two sizes only — pick by **context**, not preference. Don't mix sizes within a
single form. Addons (prefix label, suffix label, button, dropdown) match the
field's size — never combine an `md` input with an `sm` button.

| Size | Height | Padding-x | Text   | Mono text  | Use for                                                |
|------|--------|-----------|--------|------------|--------------------------------------------------------|
| `md` | `h-9`  | `px-3`    | `14px` | `13px`     | Primary forms, settings pages, modal dialogs (default) |
| `sm` | `h-7`  | `px-2.5`  | `12px` | `12.5px`*  | Sidebars, metadata strips, toolbars, inline filters    |

\* `sm` mono content uses `text-[12.5px]` (not `12px`) to match the optical
weight of surrounding sans content. This is the only off-scale size and is
reserved for compact mono-in-form contexts.

**Shared rules across both sizes:**

- Border: `border border-line` (1px). Focus darkens to `border-ink-900` — no
  thickening, no glow, no ring.
- Radius: `rounded-md` (4px) for both sizes; addon ends use `rounded-l-md` /
  `rounded-r-md`.
- Addon vocabulary: `border border-line bg-surfaceMuted` for static labels
  (prefix, suffix, prompt `$`); `border border-line bg-surface` for actionable
  buttons (Copy, dropdown, file picker). Hover lifts to `bg-surfaceMuted` →
  `bg-ink-300` only on actionable addons.
- Disabled state shares the form vocabulary: `border-dashed border-line`,
  `bg-surfaceMuted`, `text-ink-400`, `opacity-70`.
- Focus ring on actionable addon buttons: `focus-visible:outline outline-2
  outline-offset-[-2px] outline-ink-900` (inset so it doesn't break the group's
  rounded corners).

### Motion & Effects

Motion tokens are explicit and reusable. Default to `base + standard`; pick
another token only when intent demands it.

| Token      | Value                                  | Usage                          |
|------------|----------------------------------------|--------------------------------|
| `fast`     | 120ms                                  | Hover state, tooltip in        |
| `base`     | 180ms                                  | Default UI transitions         |
| `slow`     | 260ms                                  | Modal/sheet enter, large moves |
| `page`     | 360ms                                  | Page-level transitions         |
| `standard` | `cubic-bezier(0.2, 0, 0, 1)`           | Default easing                 |
| `entrance` | `cubic-bezier(0, 0, 0, 1)`             | Element entering view          |
| `exit`     | `cubic-bezier(0.4, 0, 1, 1)`           | Element leaving (60–80% of in) |
| `spring`   | `cubic-bezier(0.34, 1.56, 0.64, 1)`    | Playful, sparing               |

- Animate `transform` and `opacity` only. Avoid `width`, `height`, `top`.
- Honor `prefers-reduced-motion` — collapse to instant or 60ms cross-fade.
- Shadows are restrained: `shadow-tooltip` for floating UI, `shadow-card` for
  the elevated card pattern. Dark mode adds a 6% inset top highlight so cards
  feel lifted off the dark paper instead of submerged into it.
- `::selection` uses `accent` at 35% via `color-mix(in oklab, ...)`.
- Border radius scale: `2px` (sm), `3px` (default), `4px` (md), `5px` (lg),
  `9999px` (pill). Rounded but not soft — buttons/cards stay editorial.

### Design Principles

1. **Content over chrome** — Every pixel should serve a purpose. No decorative
   elements that don't aid comprehension. Package data is the interface.

2. **Speed is a feature** — The UI should feel instant. Server-side rendering,
   minimal JavaScript, aggressive caching, progress indicators that show real
   work happening. No loading spinners hiding slow backends.

3. **Terminal-native** — The CLI and TUI are first-class interfaces, not
   afterthoughts. Respect `NO_COLOR`, use standard terminal conventions, and
   prefer information density over visual embellishment.

4. **Consistency across surfaces** — The same package should feel recognizable
   whether viewed in the CLI, TUI, or web frontend. Use the same naming
   conventions (`namespace:name`), the same data hierarchy, and the same
   accent color where applicable.

5. **Accessible by default** — Sufficient color contrast (4.5:1 minimum for all
   body text and ink-400 detail), semantic HTML, keyboard navigation, focus
   rings on `accent`, respect for user preferences (color, motion).

### Learned Preferences

- **Strip, don't add.** When something feels wrong, the answer is almost always
  to remove elements rather than add new ones. Less is more — always.
- **No chrome.** Background colors on nav bars, colored headers, decorative
  borders — all rejected. Nav floats: just a wordmark and a few text links
  at the top. No background, no border, no visual weight.
- **Hierarchy through spacing, not decoration.** Sections are separated by
  generous whitespace, not borders or background bands. Group related elements
  tightly and separate groups with breathing room. Editorial 1.5px `rule`
  dividers reserved for true section breaks.
- **Text links over buttons for secondary actions.** Only primary actions get
  button treatment. Visual weight proportional to importance.
- **Cards over rows.** Listings use a responsive card grid (1 / 2 / 3 columns).
- **Search is the hero.** Autofocused, prominent, with the package count baked
  into the placeholder text. Everything else supports discovery.
- **Monospace for identity.** Package names (`wasi:http`), commands
  (`wasm install`), versions, durations, and IDs use mono. Not decoration —
  it signals "this is something the system or user types."
- **Inline code is a chip, not bare text.** Wrap it in a `<code>` with a
  `bg-surfaceMuted` background and slight padding. Bare colored mono mid-sentence
  causes brightness jumps that read as missing words, especially in dark mode.
- **Both themes are tuned, not inverted.** Pastels gain chroma in dark mode
  (~2x) so hue identity survives the dark canvas. Semantics get pulled to
  L≈0.78 in dark to feel restrained instead of neon. Never let `bg-ink-900`
  carry through to scrims or overlays — those need fixed dark values.
- **`color-mix()` over Tailwind opacity for `var()` colors.** Tailwind's
  `bg-foo/85` modifier silently fails on CSS variables. Use
  `color-mix(in oklab, var(--c-x) 85%, transparent)` inline.
- **Categoricals are indexed, not semantic.** Pick `cat-blue` because it's
  position 1, not because blue means anything specific. Reserve `positive`,
  `negative`, `accent` for actual semantic meaning.
