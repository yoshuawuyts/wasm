Configuring the registry

The wasm.toml manifest lives at the root of every package. It declares the package identity, its dependencies, and the registries it pulls from.

Manifest fields

  • name — reverse-DNS package identifier

  • version — semantic version, must be unique per registry

  • authors — one or more contact strings

Resolution order

  1. Local cache at ~/.wasm/store

  2. Registries declared in the manifest, in order

  3. The default registry, unless --offline is set

Registries are append-only. A version, once published, cannot be overwritten — only yanked. 
[package]
name    = "example.com/hello-world"
version = "0.1.0"
authors = ["Lorem Ipsum <lorem@example.com>"]

[dependencies]
"wasi:http"   = "0.2"
"wasi:cli"    = "0.2"

See the manifest reference for the complete schema, including optional fields like [targets.*] and [features].

Code Samples

One panel — pre.id-code — sitting on --c-surface, with token colours pulled from the theme-aware --color-wit-* palette so chroma stays balanced on both light and dark pages. Three forms: a plain block, a tabbed multi-language block, and a paired request / response grid.

Plain · single panel
[package]
name    = "example.com/hello-world"
version = "0.1.0"
authors = ["Lorem Ipsum <lorem@example.com>"]

[dependencies]
"wasi:http"   = "0.2"
"wasi:cli"    = "0.2"

Default form. Used for one-off TOML, JSON, or shell snippets where the language is obvious from context.

Tabbed · same call, multiple languages

Tab strip uses .id-lang-tabs + .id-lang-tab. Active tab fuses with the panel (shared surface, top-left radius zeroed). The .is-soon modifier dims tabs whose bindings haven't landed yet — users see where future languages will appear without the link being live.

Two panels in a 2-col grid; each carries its own 11px ink-500 mono uppercase label above. Use for HTTP / RPC reference where the response shape is part of the contract. Stacks to single column on narrow viewports.

Each token references the same --color-wit-* variables we use for WIT diagrams — module pink, world purple, interface sky, function green, struct indigo, resource amber. Both light and dark themes carry calibrated hex pairs so the panel never feels over-saturated on paper or muddy on graphite.

Anatomy

  • Quiet inset panel — --c-surface background, --c-line-soft hairline border, 5px radius. Inverts with the rest of the page in dark mode.

  • 12.5px monospace, line-height 1.55, padding 14×16. Horizontal overflow scrolls rather than wrapping — code never reflows.

  • Tokens are short class names on <span>: .k, .s, .n, .ty, .fn, .at, .h, .v, .f, .p, .c. Untagged characters fall back to the panel's default --c-ink-900.

  • Caption above each panel uses 12px ink-500; request/response labels use 11px mono uppercase ink-500 with tracking-wider for parity with the rest of the system.

  • Tabbed variant: .id-lang-tabs sits flush against the panel; the active tab shares the panel's dark surface and zeros its bottom-left corner so the two read as one piece.

11

Table

Two patterns cover everything: a definition table (no <thead>, identifier on the left, meaning on the right) and a tabular table (labeled columns, tabular-nums for figures). 13px body, 1.5px soft row separators (border-lineSoft), py-3 rows. When the leading column is a category, use the .id-http-status pill family.

Definition · identifier and meaning
version string · required SemVer tag, e.g. 0.2.4.
signature string · optional Detached cosign-compatible signature, base64 encoded.
--scope user project system Where to write the registry entry. Defaults to user.
--config <path> path Override the project config file location.

Used for request body fields, CLI flags, environment variables, schema docs — anywhere each row is "name + what it does." No <thead> because column labels would just repeat what the cells already show. Left column auto-sizes with whitespace-nowrap: mono identifier in ink-900 (with optional <placeholder> in ink-400), then a meta slot on the same baseline holding either an 11px ink-500 qualifier (string · required, path) or enum values as small mono chips (bg-surfaceMuted px-1 py-0.5 rounded-sm). Right column is reading-flow prose at ink-700.

Tabular · labeled columns with status and metrics
Status Code Calls Avg latency Meaning
200 ok 1 240 38 ms Request succeeded.
400 invalid_request 12 4 ms Malformed parameter or request body.
401 unauthenticated 0 — Missing or invalid bearer token.
500 internal_error 1 612 ms Unexpected server-side failure.

Used when columns warrant labels — error / response code reference, telemetry, anything with multiple typed columns. Headers are quiet: text-ink-400 font-normal at the body type size, same case as the data — they label columns without shouting. The first body row's border-t-[1.5px] border-lineSoft is the only rule between header and body, identical to the row separators below it. Numeric columns get tabular-nums text-right; zeros and N/A drop to ink-400; negatives or out-of-band values use text-negative; totals use font-medium. Categorical leading columns use the .id-http-status pill family in a fixed-width column so the next column aligns down the page.

Inputs sit on a surface with a 1px line border. Focus darkens the border to ink — no thickening, no glow. Two sizes: md (default) for primary forms, sm for dense contexts like sidebars, metadata strips, and toolbars.

Grid

Pages live in a max-w-[1440px] container with px-4 md:px-6 gutters. Inside, a small set of column shapes covers every layout: three-column (sidebar · reading · on-this-page) for documentation; two-column for narrative pages and this style guide; single column bounded by a reading measure for prose. Reading text is always capped at max-w-[72ch] regardless of the column it sits in.

Three-column · sidebar · reading · on this page
240px
1fr · max-w-[72ch]
200px
<div class="max-w-[1440px] px-4 md:px-6 grid grid-cols-1
            lg:grid-cols-[240px_1fr_200px] gap-8 lg:gap-10 pt-8 pb-24">

Reference layout in docs.html. Three concrete column widths (240px / 1fr / 200px) — the sidebar carries the full nav tree, the centre column reads, the right column lists in-page anchors. Side columns use lg:sticky lg:top-16 lg:self-start lg:max-h-[calc(100vh-5rem)] lg:overflow-y-auto so they pin once the page scrolls past the header. Below lg the layout collapses to grid-cols-1; the right column is hidden (hidden lg:block), the left becomes a standard top-of-page nav. Gutters: gap-8 lg:gap-10 (32 → 40px) — generous because three regions need air to read as separate concerns.

Two-column · section label · content
200px
1fr
<div class="grid md:grid-cols-[200px_1fr] gap-6 md:gap-12">

The pattern this style guide uses for every section. 200px for the section label / synopsis; 1fr for the demos. Wider gutter (md:gap-12 = 48px) than the three-column layout because there are only two columns to space.

Single column · reading measure
max-w-[72ch]
<article class="prose max-w-[72ch] mx-auto">

Long-form prose, blog posts, README-style pages. 72ch is the body-copy reading measure (~640px at 14px); larger and the eye loses the line. Even inside the three-column grid, the reading column constrains its content to max-w-[72ch] — the column gets the available space, the prose doesn't fill it.

Rules

  • 01

    Outer container is always max-w-[1440px] px-4 md:px-6. Use full bleed (no max-w) only for region surface swaps.

  • 02

    Pick the widest layout that fits the content, not the page. A docs page with no in-page anchors uses the two-column layout, not three-column with an empty column.

  • 03

    Side columns are fixed-width (240px, 200px); only the centre column is 1fr. This keeps the reading column stable as the viewport grows.

  • 04

    Gutter scales with column count: two columns get md:gap-12 (48px), three columns get lg:gap-10 (40px). More columns means more boundaries, so each gets less air.

  • 05

    Reading text is always max-w-[72ch] inside its column. Demos, code, and tables can fill the column.

  • 06

    Side columns become sticky at lg: and above with lg:sticky lg:top-16 lg:self-start lg:max-h-[calc(100vh-5rem)] lg:overflow-y-auto. Below that breakpoint they collapse into the document flow.

  • 07

    The right column is the first to drop on narrow viewports (hidden lg:block). On-this-page is a navigation aid, not load-bearing.

Regions

Pages are composed of stacked regions. The primary region uses the canvas surface; secondary regions (supporting data, references, appendices) switch to the white surface. The surface swap signals “this is additional content” — no rules or borders are drawn between regions.

Rules

  • 01

    The primary region (top of page) sits on canvas. Use it for the main subject.

  • 02

    Secondary regions sit on surface. The boundary is the surface swap itself — no rule, no border.

  • 03

    Use full-bleed background swap on wide screens so the boundary reads as a true section break, not a card.

Motion

Motion is functional: it explains state changes, never decorates them. Most transitions sit between 120–260ms on the standard curve. Anything longer needs a reason.

Easing curves

Standard
cubic-bezier(.2,0,0,1)
Default for state changes — hover, focus, expand.
Entrance
cubic-bezier(0,0,0,1)
Elements arriving on screen — toasts, popovers, modals.
Exit
cubic-bezier(.4,0,1,1)
Elements leaving — dismissed alerts, closed sheets.
Spring
cubic-bezier(.34,1.56,.64,1)
Reserved for direct manipulation feedback — toggles, drag-snap.

Durations

fast120msColor, opacity, focus rings.
base180msDefault transition. Hover, expand-collapse.
slow260msPosition changes, panel slides.
page360msRoute transitions, modal open.

Preview (hover row)

fast · std
base · std
slow · std
page · spring

Rules

  • 01

    Animate transform and opacity only. Avoid width, height, top.

  • 02

    Default to base + standard. Pick another token only when intent demands it.

  • 03

    Pair entrance and exit asymmetrically: enter on entrance, leave on exit, exits 60–80% of the entrance duration.

  • 04

    Spring is for human-initiated, direct manipulation only. Never for system events.

  • 05

    Honor prefers-reduced-motion — collapse to instant or to a 60ms cross-fade.

  • Name is 13.5px mono ink-900 medium; underlines on hover only. Description is 13px ink-700, one line, no wrapping pressure.

  • Trailing .meta stays mono ink-500 — version, status, or count. Drop it entirely if there's nothing to show.

  • Sigil colour follows the C01 convention so the same kind reads consistently across sidebar and item list.

  • Hover tints the entire row to --c-surface-muted — the only affordance that the row is interactive. No border or accent change.

  • Deprecated rows fade name + description to ink-400, strike through the name with a 1px line, and drop the sigil opacity to 50%. Set the trailing .meta to a one-word state (“deprecated”, “removed”) instead of a version.

Item Details

A method/kind pill anchors the symbol below the title; a one-sentence tagline explains it; an optional structured request-body table, a responses list, and paired example panels stack below in fixed order. Used as the destination from Item List rows.

Cornerstone · POST with auth and request body

v1.4.0 source spec

Push a new version of a package to a registry. The body references an OCI manifest already uploaded via the standard distribution endpoints.

Anatomy

  • Container — the whole article sits inside a rounded-lg border border-line bg-canvas card with p-5 md:p-6 padding, mirroring the bordered surfaces used by the search trigger and the navbar mockups. The card frames the symbol so it reads as a discrete reference unit, not loose page chrome.

  • Title — page-level <h2> in .id-title-head with an aux row on the right (.id-since-tag for the version a method/endpoint was introduced, plus .id-src-link entries for source / spec / man). Always the first element on the page.

  • Header (signature) — sits directly below the title so readers see what the symbol is before reading what it does. Pill (.id-method for HTTP, .id-kind for RPC) on the left, mono path (.id-path) center, optional .id-auth-tag pushed right via margin-left: auto. Hairline-bordered band on canvas — distinct from cards, anchored to the article.

  • Tagline — a single-paragraph .id-page-tagline below the signature, capped at 72ch. What this thing does, in one sentence. Acts as the bridge from the signature to the structured sections below.

  • Path coloring — static segments stay ink-900, parameters ({registry}) take plum-ink so the eye can scan placeholders without reading every character. Slashes drop to ink-400 as low-contrast punctuation.

  • Request body (optional) — structured table, not a code block. Each row: a name + type/required line on the left (w-[180px], mono name + 11.5px ink-500 mono meta) and a description on the right. Hairline rules between rows. Use this whenever the body has more than two fields or any optional fields — code blocks become hard to scan past ~3 keys.

  • Responses — divided list. HTTP uses .id-http-status pills (2xx green / 3xx blue / 4xx peach / 5xx pink). RPC uses .id-status-dot + fixed-width mono code (w-20 so descriptions align).

  • Example — for HTTP, paired request/response code panels in a 2-col grid. For RPC, language tabs above a single panel; inactive tabs are is-soon placeholders so users see where future bindings will land.

  • Sections always appear in this order: title → header → tagline → request body → responses → example. Optional sections (request body, auth tag, example, since-tag, source/spec links) drop cleanly without reordering or backfilling — predictability across pages matters more than local optimization.
