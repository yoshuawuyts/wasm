§ 01

Overview

wasi:http is the standard wire-format-free HTTP interface for WebAssembly components. It splits into two halves: incoming requests delivered to a guest, and outgoing requests issued from a guest. Both share a common types interface for headers, bodies, and streams.

Package declaration
package wasi:http@0.2.4;

/// Portable HTTP for WebAssembly components.
interface types { ... }
interface incoming-handler { ... }
interface outgoing-handler { ... }

world proxy {
  import types;
  import outgoing-handler;
  export incoming-handler;
}

§ 04

world proxy §

since 0.2.0 source spec

The canonical HTTP component shape: imports the types and outgoing-handler interfaces from the host, and exports incoming-handler back. Hosts instantiate one component per request.

WORLD wasi:http/proxy Stable
Definition
world proxy {
  import wasi:cli/stdout;
  import wasi:cli/stderr;
  import wasi:clocks/wall-clock;
  import wasi:clocks/monotonic-clock;
  import wasi:random/random;

  import types;
  import outgoing-handler;

  export incoming-handler;
}
Imports
types interface Shared HTTP types (request, response, fields, body streams).
outgoing-handler interface Lets the component send HTTP requests through the host.
wasi:clocks/* interface · 2 Wall and monotonic clocks for timeouts and timestamps.
Exports
incoming-handler interface · required The component must supply handle.
Example · Rust
use wasi::http::types::{IncomingRequest, ResponseOutparam};

struct MyProxy;

impl wasi::exports::http::incoming_handler::Guest for MyProxy {
  fn handle(req: IncomingRequest, out: ResponseOutparam) {
    // build a 200 OK and hand it back to the host
    let resp = OutgoingResponse::new(Fields::new());
    ResponseOutparam::set(out, Ok(resp));
  }
}

wasi::http::proxy::export!(MyProxy with_types_in wasi);

interface incoming-handler §

since 0.2.0 source

Exported by any component that handles HTTP requests on behalf of a host. The host hands the component an incoming-request and a response slot, and the component fills the slot before returning.

§ 06

func handle §

since 0.2.0 source

Process a single HTTP request. The function does not return a response value — instead, the implementation writes the response (or an error) into the response-outparam. This makes streaming responses cheap: the body can be produced asynchronously after the function returns.

FUNC incoming-handler.handle Stable
Signature
handle: func(
  request:      incoming-request,
  response-out: response-outparam,
);
Parameters
request incoming-request · owned The request handed in by the host. Ownership transfers to the guest; drop to release host resources.
response-out response-outparam · owned A write-once slot the implementation must fulfil with either an outgoing-response or an error-code.
Returns

Nothing. Side effects flow through response-out. Failing to fulfil the outparam before the function returns is a host-defined trap — typically materialised as a 500 by the runtime.

Rust
fn handle(req: IncomingRequest, out: ResponseOutparam) {
  let hdrs = Fields::from_list(&[
    ("content-type".into(), b"text/plain".into())
  ]).unwrap();
  let resp = OutgoingResponse::new(hdrs);

  let body = resp.body().unwrap();
  body.write().unwrap().write(b"hello, wasm\n").unwrap();
  OutgoingBody::finish(body, None).unwrap();

  ResponseOutparam::set(out, Ok(resp));
}
JavaScript
export const incomingHandler = {
  handle(req, out) {
    const hdrs = new Fields([
      ["content-type", "text/plain"]
    ]);
    const resp = new OutgoingResponse(hdrs);

    const body = resp.body();
    body.write().write(
      new TextEncoder().encode("hello, wasm\n")
    );
    OutgoingBody.finish(body);

    ResponseOutparam.set(out, { tag: "ok", val: resp });
  }
};

§ 08

variant method §

since 0.2.0 source

The HTTP method of a request. Standard methods are spelled out as nullary cases; non-standard verbs are carried in the other case as a string.

VARIANT types.method Stable
Definition
variant method {
  get,
  head,
  post,
  put,
  delete,
  connect,
  options,
  trace,
  patch,
  other(string),
}
Cases
  • getIdempotent retrieval. No body.
  • headSame as GET, headers only.
  • postSubmit data. Not necessarily idempotent.
  • putReplace a resource at the target URI.
  • deleteRemove the target resource.
  • patchApply a partial modification.
  • other(string) Any non-standard method. The string must be a valid token per RFC 7230.
§ 09

variant error-code §

Experimental source

Structured error returned by handler operations. Cases mirror the shape of network failures rather than HTTP status codes — application-level errors should still be reported as a normal response with the appropriate status.

VARIANT types.error-code Experimental
Selected cases
  • DNS-error(dns-error-payload) Name resolution failed. Carries the underlying rcode.
  • connection-refused Peer actively refused the connection.
  • connection-timeout Connect deadline elapsed without a TCP handshake.
  • TLS-protocol-error The TLS handshake failed for protocol reasons.
  • HTTP-request-denied Host policy refused to issue the request (sandbox / capability).
  • internal-error(option<string>) Catch-all. Implementations should prefer a more specific case.
